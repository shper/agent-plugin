# refine 形态 — 精炼（一形态两方向）

> 串行精炼链：双声部各自生成或单向生成→质检，主裁收口（合并 / 修订）。
> 两方向同源，靠 `--direction` 参数分流。共享规范（宿主无关 §0 / 底座 §1 / 触发 §6 / 落盘 §7 / 外部接入 §8 / 降级 §9）见 `./consult-common.md`。

---

## 1. 方向选择（two-way 默认）

| 方向 | 拓扑 | 适用 | 触发措辞 |
|---|---|---|---|
| **`two-way`**（默认） | 独立生成 → 交叉互评 → 主裁合并 | 方案 / PRD 草稿 / 文案的多视角打磨 | "互评 / 多视角打磨 / 双人对照" |
| **`one-way`** | 生成 → 单向质检 → 主裁修订 | 报告 / 文档 / 方案草稿的生成-质检-润色（含「仅质检」旁路） | "质检 / 审一下这篇 / 润色 / 把关" |

入参 `--direction two-way|one-way`（默认 `two-way`）。`one-way` 须有"待审材料/草稿"——无材料则先用 `two-way` 产出（consult-common §6 形态适配检查）。

## 2. `two-way`（双向互评 → 合并）

### 2.1 拓扑

| 步骤 | 角色 | 底座 | 产出 |
|---|---|---|---|
| 1 独立生成（**可并行**） | Agent A / Agent B | external_voices[host][0] / [1] | 同任务各出初版 |
| 2 交叉互评（串行） | A 评 B → B 评 A | 同上 | 逐条 **H/M/L + 评分(0–100)**，整体附质量分 |
| 3 合并 | 主裁 = 当前宿主主模型 | 主会话 | 终稿 + 来源标注 + 差异摘要 |

### 2.2 评分门控

双方互评均无 H 且综合 ≥85 → 视为无实质问题，**跳过独立修订直接合并**。

### 2.3 收口契约

| 收口动作 | 固定输出 |
|---|---|
| **合并** | 合并终稿（**标注每段来源 A/B/综合**）+ 对 M/L 争议的裁定 + 关键差异摘要；若双方无 H 且 ≥85 则免独立修订直接合并 |

## 3. `one-way`（单向质检 → 修订）

### 3.1 拓扑

| 步骤 | 角色 | 底座 | 产出 |
|---|---|---|---|
| 1 生成 | 生成者 | external_voices[host][0] | 按输入生成初版（**`--skip-gen` 旁路**：用户已有内容时跳过，直接进质检） |
| 2 质检 | 质检者（**须 ≠ 生成底座**） | external_voices[host][1] | 逐条 **H/M/L** + **整体评分(0–100)** |
| 3 修订 | 主裁 = 当前宿主主模型 | 主会话 | 修订版 + 折叠质检报告 + 逐条处理说明 |

### 3.2 H/M/L 标准

| 等级 | 标准 |
|---|---|
| **H** | 事实错 / 逻辑矛盾 / 严重遗漏（**全改**） |
| **M** | 表达 / 结构 / 完整性（**尽量改**） |
| **L** | 无问题（**保持**） |

### 3.3 评分门控

整体评分 ≥85 → **跳过修订**，直接输出初版为终版（注明"质检免修订"）；<85 → 进入修订。

### 3.4 收口契约

| 收口动作 | 固定输出 |
|---|---|
| **修订** | 修订版（H 全改 / M 尽量 / L 保持）+ 折叠质检报告 + 逐条处理说明；若质检 ≥85 则原样输出初版并注明"质检免修订" |

## 4. 角色 prompt 骨架

外部模型的 prompt（主会话喂给 `orchestrate.py`，按方向填角色）：

```
你是多模型协作的【Agent A / Agent B / 生成者 / 质检者】。任务：<自包含上下文>。
立场与产出要求：
  - two-way 独立生成：按任务出初版，不参考另一方（你看不到 B 的版本）；
  - two-way 互评：逐条 H/M/L + 评分(0–100)，附整体质量分；
  - one-way 生成：按任务出初版；
  - one-way 质检：逐条 H/M/L + 整体评分（H 事实错/逻辑矛盾/严重遗漏；M 表达/结构/完整性；L 无问题）。
```

> 所有外部 transport 的只读封装在 `ai_client/providers.py`（见 consult-common §8）。

## 5. 物理架构（refine 专属）

**走 `orchestrate.py` 不进 `panel.js` 的理由**：①refine **跨底座**（Workflow.agent() 调不了外部底座）；②有**串行依赖**（生成→互评/质检），靠主会话逐步调模型不可复现。`orchestrate.py` 用 asyncio 把外部底座多步拓扑确定性编排成结构化 JSON（`steps` 含各步 text/error，refine 还带 `direction`）。

**收口（合并/修订 = 宿主主裁）**调不了主会话自己，留主会话——脚本跑到收口前即止，只输出素材（双初版+互评 / 初版+质检评分）。各步外部调用的请求+生响应由脚本**自动留痕**（consult-common §7.2）。

## 6. 编排骨架

### 6.1 一次性确定性编排

承接：留痕会话 `$TASK` 已建立（SKILL.md Step 5 / consult-common §7.2），外部声音池 `external_voices[host]` 已取定。

```bash
ROOT="${CLAUDE_PLUGIN_ROOT:-$PLUGIN_ROOT}"   # 变量在 hook 外常为空→据本 skill 目录上两级代入插件根（consult-common §3）
uv run "$ROOT/ai_client/orchestrate.py" refine \
  --task "$TASK" \
  --ext0 <ext0> --ext1 <ext1> \
  --fallback <宿主底座 provider> \
  "<任务>" \
  --direction two-way|one-way \
  [--context "<自包含背景>"] [--file <doc>] [--skip-gen]
```

`<ext0>`/`<ext1>` = `external_voices[host]` 两席（refine `two-way`=A/B、`one-way`=生成/质检）。`--skip-gen` **仅 `one-way` 生效**——已有草稿时直接进质检。`--fallback` 传**宿主底座** provider：某外部步骤失败时脚本**确定性补位**重试、打 `degraded` 标注（consult-common §9）。注：`one-way` 质检者须 ≠ 生成者底座，若两者都降级到同一 fallback 则质检失去独立性，主裁应据 `degraded` 标注折扣或流产。

### 6.2 主裁收口（接 SKILL.md Step 6）

承接：`orchestrate.py` 输出的 JSON 已得到。

| 方向 | 主裁动作 | 门控 |
|---|---|---|
| `two-way` | 读两初版 + 互评 → §2.3 合并契约 | 双方无 H 且 ≥85 → 跳独立修订直接合并 |
| `one-way` | 读初版 + 质检评分 → §3.4 修订契约 | 质检 ≥85 → 免修订直接输出初版 |

任何步 `error`/`skipped` → 按 consult-common §9 降级：

- 外部只剩 1 个 → 一方用外部、另一方用宿主底座补位，标"<角色>与收口同底座"；
- 收口角色（合并/修订 = 主裁）失败 → 输出已收集素材让用户判断，**不把收口降级到对抗方同实例**。

收口结论按 SKILL.md Step 8 写入留痕（`consult_log.py verdict --mode refine`）。
