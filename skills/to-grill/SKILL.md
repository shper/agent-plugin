# to-grill — 需求审问 / 讨论

通过**一次一个问题**的 grill 式审问，把模糊的想法、方案、决策树逼到清楚、达成共识。

> 定位：需求/方案阶段的审问引擎，可在任意点切入——审问已有的 backlog / PRD / spec 类草稿文件，或当前对话里的想法。
> 边界：**只回写被审问的目标文件本体**；不另外沉淀新文件、不产新 PRD、不改代码、不碰宿主项目其它文档（决策外溢交宿主项目对应的下游同步工具，若已安装）。

## 两种输入 + 落盘分场

| 输入 | 审问对象 | 落盘 |
|------|---------|------|
| **传入草稿文件路径** | 该文件 + 项目可选的词汇表 / 架构决策（若存在）+ 相关代码 | **回写该文件本体**：在文件内以适合该类型的方式追加 `决策（YYYY-MM-DD）` 段；如该文件有"决策登记表 / 状态字段"等既有结构，按其结构填入；按宿主项目元数据维护纪律（如有）同步索引/变更日志（不确定就主动征询用户） |
| **无参 / 审问对话内容** | 当前对话里的想法、方案 | **不落盘**：讨论至收敛即止，结论交宿主项目下游工具（如装了记录/起草类 skill） |

## 通用原则

- 执行前加载必要上下文（按宿主项目约定的上下文加载纪律，如有），把议题相关背景读够；不假设有特定目录结构。
- 沿用项目既有术语，尊重并不冲突于项目已记录的架构决策（如有）。

## 审问技法

### 内核（两种输入都用）

- **一次一个问题**，等用户回答再问下一个，不一次性抛 checklist。
- **走决策树每个分支**，逐个解决决策之间的依赖。
- 每个问题**给出你的推荐答案/方案，并配详细说明**，让用户在有锚点的基础上拍板。每个推荐至少讲清：
  - **是什么**：方案的具体内容，不止抛结论。
  - **为什么**：推荐它的理由、依据（命中的代码/文档/约定，或背后的权衡逻辑）。
  - **优缺点 / 弊端 / 代价**：这个选择好在哪、牺牲了什么、有何风险或副作用、什么前提下不成立。
  - **备选对比**（有真权衡时必给）：列出 ≥1 个被否的备选，说明为何不选它——让用户看到取舍而非单选项。
  - **业界 / 先例对照**：所选相对业界成熟做法 / 项目既有先例处于什么位置（沿用 / 偏离 / 重造轮子）；偏离要给出值得偏离的理由。⚠️ 注明这是判断而非权威，能查证就查证，不硬凑"最佳实践"。
  - **最强反对（pre-mortem）**：给一条反对【所选方案本身】的最强论据（假设它日后被推翻，最可能因为什么），而非只论证备选更差。
  - 信息不足以推荐时，**说明缺什么、为什么暂时给不出推荐**，而非含糊带过。
- **选项化提问必带"拿不准"出口**：当问题以备选项形式抛给用户（A / B / C…，含推荐项），**末位固定加一项「🤔 拿不准，先会诊再给选项」**。用户选它 → 按下方「多模型会诊」点燃引擎（默认 panel，二元选型升 debate），主裁收口后**带着会诊结论重新收敛出更聚焦的选项**再问一遍（可能砍掉劣选、补出新选项、改写推荐），**不把原选项原样重抛**。非选项形式的常规单问不强加此项；能查代码/文档即答的低风险分支也别为加这项而硬开会诊。
- **收敛条件**：决策树各分支 resolved、达成共识即停——**不为凑问题数量而问**。

### 域意识（审问已有文件必用；审问对话适用）

- **术语挑战**：如宿主项目有权威词汇表（如 `GLOSSARY.md` 之类），以其为对齐基准——用户用词命中"应避免"列表、或与既有定义冲突时，**当场引用该条目要求二选一，不跳过此环节**。无词汇表则只做通用语义清晰度挑战。
- **锐化模糊语言**：遇到含糊/过载术语，提一个精确的 canonical 术语，**优先沿用项目既有词汇，不另造**；确属新概念则提示用户按其项目维护纪律记录（如有）。
- **场景压测**：讨论概念关系时造具体边界场景，逼用户把边界讲清。
- **代码 / 架构对照**：用户陈述与项目已记录架构决策（如有）或实际代码矛盾时，把矛盾摆出来让其澄清。

### 多模型会诊（可选增援）

审问到某决策分支**命中真权衡（≥2 方案取舍不明）或高风险**（写入/认证/金额，或要动项目已记录的重大架构决策）、**用户在选项题里选了末位「拿不准，先会诊再给选项」**、或用户显式要求时，点燃多模型协作引擎引入第二个声音，按 `$ROOT/skills/to-consult/SKILL.md` 执行（`$ROOT` 按 consult-common §3 三档解析：`CLAUDE_PLUGIN_ROOT`/`PLUGIN_ROOT` 有值就用、否则主会话代入 realpath 本 SKILL.md 目录上两级的绝对路径，**勿留空串**）：默认 panel 形态——宿主 persona 批（架构红队/价值质询/唱反调）走 `Workflow({scriptPath: "<$ROOT 解析后的绝对路径>/scripts/panel.js", args})`（Claude Code 专属 Workflow；**scriptPath 不展开环境变量，须先把根解析成绝对路径再传**，见 to-consult/consult-common §3；非 Claude 宿主主会话自扮演），并发 Bash 调 `$ROOT/ai_client/cli.py` 取按宿主取的外部声音池，**当前宿主主裁**摆矛盾综合后**并回审问**继续逐问；二元对立选型可升级 `mode=debate`（详见 `$ROOT/skills/to-consult/mode-debate.md`）。**克制使用**——术语对齐、能查代码即答的低风险分支不开（`to-consult/consult-common.md §6`）。引擎只产素材，结论的落盘仍按下方 Step 3 回写被审问文件本体。

## 执行步骤

### Step 1: 判定输入 + 加载上下文

- 有文件路径入参 → 复审模式：读入目标文件全文；如宿主项目有词汇表 / 架构决策库（路径由用户或项目约定提供，本 skill 不假设固定路径），一并加载作对照基准。
- 无参 → 审问当前对话内容。

### Step 2: 逐问审问

按上方「审问技法」逐问深挖，直到决策树各分支收敛、达成共识。

### Step 3: 按场景收尾

- **复审已有文件**：把结论回写目标文件本体——
  - 若该文件已有"决策登记 / 状态字段 / 章节模板"等既有结构，按其结构填入（具体格式由项目自定，本 skill 不强制特定 schema）。
  - 否则在文件末尾追加 `**决策（YYYY-MM-DD）**` 决策块，列出本次审问拍下来的结论。
  - 按宿主项目元数据维护纪律（如有）同步该文件 frontmatter 的 `updated` 等字段、所在目录索引、变更日志；**不确定就主动征询用户**而非自行猜测。
- **审问对话内容**：不落盘。把收敛后的结论小结给用户，并提示交宿主项目对应的记录/起草类下游工具（如已安装）。

## 注意事项

- **审问对话不落盘**，复审已有文件**只回写该文件本体**。
- **不另外沉淀新文件**（记录/起草交宿主项目对应下游工具），**不改代码**。
- **决策外溢**（影响项目级上下文 / 架构 / 任务总表等）：交宿主项目对应的同步工具（如已安装），本 skill 不直接动这些文件。
- 审问要互动、逐问推进；**收敛即停**，不为凑数而问。
- 每个推荐答案/方案都要**带详细说明 + 优缺点/弊端/取舍**（见「审问技法 · 内核」），不抛裸结论。
